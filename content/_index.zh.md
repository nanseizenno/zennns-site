# TPCA(Transition Pre-Control Architecture) / CAE-SDB 状态迁移前置控制架构

全野南政｜Nansei Zenno

本网站用于整理和公开 TPCA / CAE-SDB 状态迁移前置控制架构的基本定义、工程案例和技术札记。

TPCA / CAE-SDB 是用于状态迁移前判断与多路径控制的工程方法。  
适用对象包括制造系统、自动化执行单元、群控系统、MES / WCS、PLC / HMI、机器人单元以及数字调用治理系统。

在这些系统中，进入下一状态前通常需要确认现场条件、许可状态和执行链状态。现有控制或管理系统经常将判断结果压缩为：

OK / NG  
ready / not ready  
true / false  
pending / blocked  
允许 / 禁止  

这些结果可以用于流程继续或停止，但对现场排查、系统复盘和后续改善来说，信息量通常不足。现场还需要进一步确认：

不能进入下一状态的原因来自条件、许可还是执行链；  
相关问题属于结构缺失、动态异常还是控制边界；  
下一步应等待、重识别、回流、协调、降级、禁止进入、安全锁定还是人工确认；  
本次判定结果能否被记录、追溯和复盘。

TPCA / CAE-SDB 在目标状态进入前设置前置控制节点，对多源状态信号进行整理、映射、判定和控制输出，用于将分散在 PLC 程序、MES / WCS 流程、调度逻辑、设备握手和人工经验中的状态迁移条件显式化。

## 基本处理链条

TPCA / CAE-SDB 的基本处理链条如下：

当前状态 / 当前阶段  
→ 目标状态 / 目标阶段  
→ 多源状态信号  
→ C / A / E 状态映射  
→ S / D / B 判定  
→ CAE-SDB 判定结果  
→ 多路径控制输出  
→ 状态记录与追溯

其中，C / A / E 表示进入目标状态前需要确认的三类状态变量。  
S / D / B 表示对这些状态变量进行判定时采用的三类分析性质。

## C / A / E 状态变量

C（Condition）：条件状态。  
表示进入目标状态前，现场条件、对象条件、识别条件、数据条件、任务条件等是否成立。

A（Authority）：许可状态。  
表示系统是否允许进入目标状态，包括安全许可、上位许可、区域许可、人工确认、权限、授权、资源锁、对方设备许可等。

E（Execution Chain）：执行链状态。  
表示进入目标状态后，执行链是否能够接续。E 不等同于单体设备 ready，还包括本体设备、下游承接、替代路径、回退路径、异常排出路径、结果上传或回写链路等。

## S / D / B 判定性质

S（Structure）：结构完整性。  
用于判断所需信号、接口、映射关系、许可来源、路径和执行链边界是否已经定义、接入并可观测。

D（Dynamics）：动态时序有效性。  
用于判断信号是否超时、未刷新、抖动、冲突、延迟、不同步、低置信度或处于切换中。

B（Boundary）：控制边界。  
用于判断系统处于未完全确定或边界状态时，应采用等待、重试、重识别、回流、降级、禁止进入、人工确认或安全锁定等处理路径。

## 多路径控制输出

TPCA / CAE-SDB 的输出为多路径控制结果。  
典型控制路径包括：

允许进入  
等待  
重识别  
重采样  
重定位  
回流  
异常分流  
下游协调  
资源释放  
降级执行  
备用路径  
禁止进入  
安全锁定  
人工确认  
异常隔离  
状态记录  

这些控制路径用于支持现场执行、HMI 显示、MES / WCS 记录、异常复盘和后续改善。

## 工程价值

在自动化系统开发中，状态迁移条件、异常边界、执行链承接和判断记录，通常依赖成熟程序员或资深工程师在项目中自行归纳。

不同人员、不同项目、不同设备单元之间，设计粒度和判断标准容易不一致。程序可以运行，状态迁移条件是否完整、异常边界是否清楚、执行链是否接得住，仍然需要依赖个人经验判断。

TPCA / CAE-SDB 将这类经验整理为可标准化的前置判定设计方法，用于明确：

状态迁移条件如何定义；  
许可条件如何独立确认；  
执行链如何确认承接关系；  
异常边界如何分流；  
判断过程如何记录；  
新人如何按同一质量标准参与设计。

在自动化软件工程中，该方法可用于提高状态设计的一致性、复用性、审核效率和项目承接效率。

## 应用方向

### Case 1：群控协同层

Case 1 用于多主体群控系统中的协同停滞识别与结构分型。

代表场景包括 AGV / AMR 车队、多设备协同系统、WCS 调度系统、MES 协同停滞场景和制造现场搬送协同系统。

该方向面向“群体为什么停”的问题。  
在任务存在、设备未必故障、群体执行比例下降的情况下，通过外部可观测信号、群体指标和结构分型，识别停滞来自任务分配、资源竞争、能源补给、授权未成立，或当前信号不足导致的未确定状态。

### Case 2：物理执行单元层

Case 2 用于自动化执行单元进入目标物理执行阶段前的前置判定。

代表场景包括机器人抓取、压装、检测、搬送交接、视觉识别输送线机器人分拣、PLC / HMI 自动化执行单元等。

该方向面向“单元为什么不能进入下一阶段”的问题。  
在自动化执行单元进入目标物理执行阶段前，基于视觉、机器人、安全、输送、检测、下游承接和上位系统等多源状态信号，完成 C / A / E 映射和 S / D / B 判定，并输出允许进入、等待、重识别、回流、异常分流、禁止进入、安全锁定等控制路径。

### Case 3：数字调用治理层

Case 3 用于数字请求进入目标执行路径前的调用前置控制。

代表场景包括生成式 AI 调用、工具调用、企业知识库访问、API 网关、外部服务调用和智能体代理链等。

该方向面向“数字请求为什么不能进入目标执行路径”的问题。  
在模型推理、工具调用或外部服务执行之前，先确认请求条件、调用授权和执行就绪状态，并根据判定结果将请求分流至高成本路径、澄清路径、降级路径、等待路径、阻断路径或异常处理路径。
