# TPCA(Transition Pre-Control Architecture) / CAE-SDB 状态迁移前置控制架构

全野南政｜Nansei Zenno

本网站用于整理和公开 TPCA / CAE-SDB 状态迁移前置控制架构的基本定义、工程案例和技术札记。

TPCA / CAE-SDB 是用于状态迁移前判断与多路径控制的工程方法。  
适用对象包括制造系统、自动化执行单元、群控系统、MES / WCS、PLC / HMI、机器人单元以及数字调用治理系统。

在制造系统、自动化执行单元、群控系统和数字调用系统中，系统进入下一状态前，需要确认条件、许可和执行链是否成立。现场常见的判断结果通常被压缩为 OK / NG、ready / not ready、pending / blocked、允许 / 禁止等状态。

这些状态可以用于流程控制，但不便于进一步判断问题来源、处置路径和复盘依据。现场还需要确认：

不能进入下一状态的原因来自条件、许可还是执行链；  
相关问题属于结构缺失、动态异常还是控制边界；  
下一步应等待、重识别、回流、协调、降级、禁止进入、安全锁定还是人工确认；  
本次判定结果能否被记录、追溯和复盘。

TPCA / CAE-SDB 在目标状态进入前设置前置控制节点，对多源状态信号进行 C / A / E 映射和 S / D / B 判定，并输出对应的控制路径。


## 基本处理链条

当前状态 / 当前阶段  
→ 目标状态 / 目标阶段  
→ 多源状态信号  
→ C / A / E 状态映射  
→ S / D / B 判定  
→ 多路径控制输出  
→ 状态记录与追溯

C / A / E 表示三类状态变量：条件状态、许可状态、执行链状态。  
S / D / B 表示三类判定性质：结构完整性、动态时序有效性、控制边界。

## 工程价值

在自动化软件设计中，状态迁移条件、异常边界、执行链承接和判断记录，通常依赖成熟程序员或资深工程师在项目中自行归纳。

不同人员、不同项目、不同设备单元之间，设计粒度和判断标准容易不一致。程序可以运行，状态迁移条件是否完整、异常边界是否清楚、执行链是否接得住，仍然需要依赖个人经验判断。

TPCA / CAE-SDB 将这类经验整理为可标准化的前置判定设计方法，用于明确：

状态迁移条件如何定义；  
许可条件如何独立确认；  
执行链如何确认承接关系；  
异常边界如何分流；  
判断过程如何记录；  
新人如何按同一质量标准参与设计。

该方法可用于提高状态设计的一致性、复用性、审核效率和项目承接效率。



## 应用入口

### Case 1：群控协同层

用于多主体群控系统中的协同停滞识别与结构分型。

代表场景包括 AGV / AMR 车队、多设备协同系统、WCS 调度系统、MES 协同停滞场景和制造现场搬送协同系统。

解决：**群体为什么停。**



### Case 2：物理执行单元层

用于自动化执行单元进入目标物理执行阶段前的前置判定。

代表场景包括机器人抓取、压装、检测、搬送交接、视觉识别输送线机器人分拣、PLC / HMI 自动化执行单元等。

解决：**单元为什么不能进入下一阶段。**



### Case 3：数字调用治理层

用于数字请求进入目标执行路径前的调用前置控制。

代表场景包括生成式 AI 调用、工具调用、企业知识库访问、API 网关、外部服务调用和智能体代理链等。

解决：**数字请求为什么不能进入目标执行路径。**
