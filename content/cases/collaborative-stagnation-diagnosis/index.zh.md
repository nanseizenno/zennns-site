## 1. 现场问题

在制造现场中，MES 通常已经能够记录任务、设备、工位、物料、报警、产量和实绩。

WCS 可以记录搬送任务、车辆状态、路径状态和调度结果。  
设备系统可以记录运行、等待、报警和互锁状态。  
HMI 可以显示 ready、alarm、waiting、blocked、pending 等信息。  

但现场仍然经常出现一种情况：

设备没有明显故障，生产流程却持续停滞。  
任务已经存在，执行却无法持续推进。  
车辆在线，工位有需求，但补料没有形成连续执行。  
系统表面都有状态记录，但没人能快速说明停滞来自哪里。  
临时订单变化、资源占用、区域许可、下游承接、人员确认和能源补给交织在一起，最后只能靠人工查日志、问调度、看现场。

这说明问题已经不只是“数据有没有采集”。

真正缺少的是：

对制造现场协同停滞的结构化诊断能力。

MES 可以记录发生了什么。  
制造现场还需要进一步判断：

- 停滞来自任务、物料、站点、路径等条件是否成立；
- 停滞来自调度许可、资源锁、区域许可、安全许可或人工许可；
- 停滞来自设备、车辆、路径、缓存、下游承接、能源补给和结果回写等执行链是否能够接续；
- 当前问题属于结构缺失、动态异常，还是控制边界；
- 下一步应等待、改派、释放资源、下游协调、限流、人工确认，还是增强记录。

TPCA / CAE-SDB 的作用，是在制造现场恢复目标协同执行状态之前，把这个停滞入口拆开判断。

MES / WCS 协同停滞诊断模块通过 PCN 读取多源状态信号，先确认：

C：任务、物料、站点、路径和现场条件是否成立；  
A：调度许可、资源锁、区域许可、安全许可和人工许可是否成立；  
E：设备、车辆、路径、缓存、下游承接、能源补给和结果回写是否能够接续。  

然后进一步判断：

S：相关信号、接口、资源关系和执行链边界是否已经定义并接入；  
D：任务、设备、车辆、站点、许可和下游状态是否有效、同步、未超时；  
B：边界状态下应等待、重派、释放资源、改派缓存、限流、下游协调、人工确认还是增强记录。  

最终输出不只是一句“停了”或“waiting”，而是协同停滞的结构化诊断结果，以及对应的控制建议、记录和复盘依据。

## 2. 模块定位与 PCN 位置

本案例以“MES / WCS 协同停滞诊断模块”为对象。

该模块部署在 MES、WCS、设备系统、搬送系统和现场执行单元之间，用于在制造现场恢复目标协同执行状态之前，对协同停滞进行结构化诊断。

它面向的典型场景包括：

- AGV / AMR 群控搬送；
- 无人叉车、自动搬运和线边补料；
- 多设备协同生产；
- 工位等待、缓存区拥堵和路径资源竞争；
- 区域许可、资源锁、人工放行和下游承接不足；
- 临时订单变化、优先级切换和能源补给偏在。

这里可以把 AGV / AMR 群控搬送理解为首个代表场景，但模块对象本身是制造现场协同停滞诊断能力，而不是单一 AGV 功能。

![MES / WCS 协同停滞诊断模块示意图](/images/tpca/08-mes-wcs-collaborative-stagnation-module.png)

图：MES / WCS 协同停滞诊断模块部署在制造现场协同层，汇总 MES、WCS、AGV / AMR 车队、缓存区、加工设备、路径资源、自动门、电梯、充电区、安全许可系统和 HMI / 监控看板等多源状态信号，并通过 PCN 完成状态汇总、C/A/E 映射、S/D/B 判定、群体指标计算、停滞识别、结构分型、多路径控制输出和状态记录。

当前协同状态：

MES 中存在任务，WCS 中存在调度对象，设备、车辆、工位、缓存区和路径系统持续输出状态信号，但制造现场出现 waiting、blocked、pending、idle、拥堵、断续执行或整体吞吐下降等现象。

目标协同执行状态：

系统能够恢复到可持续推进的协同执行状态。任务能够继续派发，资源能够继续流转，关键执行链能够继续接续，停滞节点能够进入等待、改派、释放资源、下游协调、人工确认或增强记录等对应路径。

PCN 的位置在目标协同执行状态入口前。

```text
当前协同状态：任务存在 / 车辆在线 / 工位等待 / waiting / blocked / pending
        ↓
群控 PCN：恢复可持续协同执行前的前置判定
        ↓
目标协同执行状态：正常派单 / 改派 / 释放资源 / 下游协调 / 限流 / 等待 / 人工确认 / 安全锁定
```

这个位置很关键。

如果诊断发生在停滞结束之后，只能用于事后复盘。  
如果诊断发生在恢复目标协同执行状态之前，就可以直接参与控制路径选择。

## 3. 多源状态信号与 C/A/E 映射

MES / WCS 协同停滞诊断模块需要读取与协同执行状态相关的多源状态信号。

| 信号来源 | 代表性信号 |
|---|---|
| MES | 工单、任务、优先级、物料需求、工位需求、生产计划变更、人工确认、异常记录 |
| WCS / 调度系统 | 任务队列、任务分配结果、任务状态、派单结果、资源锁、路径分配、站点占用、缓存区状态 |
| AGV / AMR / 无人叉车 | 在线状态、自动模式、当前位置、空载 / 负载状态、电量、任务执行状态、等待状态、报警状态 |
| 设备 / 工位 | 工位可接收状态、设备运行状态、设备等待状态、缺料状态、完成状态、报警状态 |
| 路径与设施资源 | 窄通道、交叉口、自动门、电梯、区域通行权、缓冲区、装卸点占用状态 |
| 安全与许可系统 | 区域许可、安全许可、上位放行、人工放行、锁定状态 |
| 下游与回写链路 | 下游工位可承接状态、结果回写状态、履历记录状态、异常处理状态 |

这些信号进入 PCN 后，不直接生成诊断结论。PCN 先判断这些信号在目标协同执行状态入口前分别承担什么作用，再进入 C/A/E 映射。

| 变量域 | 判定问题 | 本案例中的代表项 |
|---|---|---|
| C：Condition 条件状态 | 任务、对象、站点、路径和执行对象是否具备进入协同执行的条件 | 任务是否存在；目标工位是否有需求；物料是否匹配；站点是否可达；车辆是否属于可调度对象；目标路径是否存在 |
| A：Authority 许可状态 | 系统是否允许当前任务或资源进入下一步协同执行 | 调度许可、资源锁释放状态、区域许可、安全许可、人工确认、优先级放行、上位放行 |
| E：Execution Chain 执行链状态 | 任务派发后，执行链是否能够继续接续 | 车辆是否可执行；路径是否可通行；站点是否可承接；缓存区是否可接收；下游工位是否可消耗；结果是否能够回写 |

这里有两个关键点。

第一，A 许可可以构成独立必要约束。

资源锁未释放、区域许可未成立、安全许可未成立、人工放行未完成或上位系统禁止放行时，即使任务存在、车辆在线、路径看起来可用，也不得推进目标协同执行。

第二，E 执行链不等于“有车可用”或“有任务存在”。

单台车在线，只能说明局部执行主体存在。  
E 还包括路径接续、站点承接、缓存承接、能源补给、下游消耗和结果回写。

```text
车辆在线 ≠ 协同执行链成立
任务存在 ≠ 协同执行链成立
```

车辆在线，但目标站点不可承接，执行链仍不成立。  
任务存在，但资源锁未释放，执行链仍不成立。  
任务存在，车辆在线，但大量车辆集中充电，可执行车辆比例下降，执行链仍不成立。  
任务存在、车辆在线、路径记录存在，但下游工位缺料或不能消耗，执行链仍不成立。

这就是很多现场“系统都在线、任务也在，但整体仍然停滞”的结构原因。

## 4. S/D/B 性质判定

S/D/B 用于判断 C/A/E 中的问题属于结构、动态还是边界。

| 判定性质 | 判定问题 | 本案例中的代表项 |
|---|---|---|
| S：Structure 结构完整性 | 需要的信号、接口、资源关系和执行链边界是否已经定义并接入 | MES 与 WCS 的任务映射是否存在；路径资源是否被纳入诊断；资源锁来源是否明确；下游承接状态是否可观测；能源补给状态是否纳入诊断 |
| D：Dynamics 动态时序有效性 | 当前任务、设备、车辆、站点、许可和路径状态是否有效、同步、未超时、未冲突 | 任务是否过期；派单结果是否未刷新；车辆位置是否延迟；站点状态是否抖动；许可是否瞬时撤销；路径占用状态是否不同步 |
| B：Boundary 控制边界 | 边界状态下应进入哪条控制路径 | 是否继续等待；是否改派缓存区；是否释放资源锁；是否限制低优先级任务；是否限制同时充电；是否人工确认；是否进入增强记录 |

S/D/B 的价值在于，它不把所有停滞都写成“waiting”或“调度异常”。

同样是现场停滞，性质可能完全不同。

- 关键站点承接状态没有接入：E-S，执行链结构问题；
- 调度结果长时间未刷新：A-D 或 E-D，动态问题；
- 路径资源短时竞争：E-B，边界问题；
- 多台车集中充电导致可执行比例下降：E-B，能源边界问题；
- 上位许可撤销：A-B，许可边界阻断；
- MES 工单变更未同步到 WCS：C-D 或 C-S，条件结构 / 动态问题。

同一个“系统停了”，进入 PCN 后会变成不同的工程处置路径。

## 5. 群体指标与停滞识别

协同停滞诊断模块不仅判断单条任务，还需要观察群体层面的停滞指标。

代表性指标包括：

- 群体未执行比例；
- 群体辅助执行比例；
- 群体能源补给比例；
- 许可未成立比例；
- 路径资源占用比例；
- 站点不可承接比例；
- 异常车辆比例；
- 关键任务超时比例。

通过这些指标，可以进一步识别代表性的协同停滞结构，例如：

- 分配约束型；
- 资源竞争型；
- 能源偏在型；
- 许可未成立型；
- 下游承接不足型；
- 未确定型停滞。

这里的重点不是把停滞简单分类，而是把群体停滞转换为后续控制与复盘可用的结构化识别结果。

例如：

任务很多，但有效派单比例持续下降，群体未执行比例持续升高，可能提示分配约束型停滞。  
车辆大量集中在交叉口、自动门、电梯或窄通道附近等待，路径资源占用比例持续偏高，可能提示资源竞争型停滞。  
大量车辆集中进入充电区，可执行车辆比例持续下降，可能提示能源偏在型停滞。  
任务存在，但目标工位不可承接比例持续升高，可能提示下游承接不足型停滞。  
指标同时表现异常，但难以归并到单一结构时，应进入未确定型停滞，并触发增强记录。

## 6. 代表性停滞路径与诊断输出

PCN 的输出应直接服务于 MES / WCS 的协同干预路径。

| 现场状态 | 前置判定结果 | 诊断输出 / 控制建议 |
|---|---|---|
| 任务存在，车辆可执行，路径可通行，目标站点可承接 | C/A/E 成立 | 允许继续派单，恢复正常协同执行 |
| 任务存在，但目标站点暂时不可承接 | E-B | 等待、下游协调、改派缓存区 |
| 资源锁未释放，后续任务无法进入 | A-B | 释放资源锁、人工确认、增强记录 |
| 多台车集中在关键路径资源附近等待 | E-B | 通行权仲裁、限流、任务改派、局部缓冲 |
| 车辆大量集中充电，可执行比例下降 | E-B | 限制同时充电、调整优先级、改派任务 |
| 许可未成立，调度对象无法继续推进 | A-B | 等待许可刷新、人工放行、禁止进入下一步 |
| MES 与 WCS 状态映射不完整 | C-S / A-S / E-S | 补充信号、补充接口、增强记录、禁止依赖该诊断项自动放行 |
| 多个指标同时异常，结构暂不明确 | B | 进入未确定型停滞，增强记录、人工确认、复盘分析 |

这些输出说明，协同停滞诊断模块的结果不只是“报停滞”，而是把停滞转化为可执行的下一步。

输出可以面向多个层面：

- MES 诊断界面展示；
- WCS 调度提示；
- 事件通知；
- 干预建议；
- 履历记录；
- 后续复盘输入。

现场人员看到的不再只是 waiting、blocked、pending，而是当前停滞更接近哪一类结构问题，以及下一步应该做什么。

## 7. 状态记录、复盘与模块价值

PCN 在输出停滞识别结果和控制建议时，应同步记录判定过程。

代表性记录内容包括：

- 当前协同状态；
- 目标协同执行状态；
- 输入信号及来源；
- 信号时间戳；
- C/A/E 映射结果；
- S/D/B 判定结果；
- 群体指标；
- 停滞识别结果；
- 控制建议或控制路径；
- 人工确认结果；
- TraceLogID；
- 后续处理结果。

这些记录可以用于：

- 现场复盘；
- 事件回顾；
- 高频停滞点识别；
- MES 诊断能力升级；
- HMI / 报表显示；
- 新人培训；
- 模板复用；
- 后续 AI 辅助分析输入。

记录的重点不是简单堆积日志，而是保存“为什么制造现场没有恢复到可持续协同执行状态”的判定结构。

本案例说明，MES / WCS 协同停滞诊断模块可以把制造现场中的协同停滞从“记录现象”推进到“解释结构”。

它带来的直接价值包括：

- 让 waiting、blocked、pending 等压缩状态得到展开；
- 让分配约束、资源竞争、能源偏在、下游承接不足、许可未成立和未确定型停滞能够区分；
- 让后续等待、改派、释放资源、限流、人工确认和增强记录等路径显式化；
- 让 MES 从“记录生产过程”进一步升级为“解释协同停滞机理”；
- 让制造现场停滞诊断具备可记录、可复盘、可复用的结构。

## 8. 公开边界

本文为公开说明案例，用于解释 TPCA / CAE-SDB 在制造现场协同停滞诊断模块中的工程逻辑。

本文不展开具体阈值、等待窗口、重试次数、控制优先级完整规则、调度算法细节、接口配置、数据库结构或完整工程实现参数。

实际项目中的具体设计，应结合 MES 平台、WCS 平台、现场对象、调度策略、安全要求和交付规范另行确定。
