TPCA / CAE-SDB 解决的是一个具体问题：系统进入目标状态之前，怎么判断条件成不成立。进不去的话，问题在哪，下一步走哪条路。

TPCA 是架构，CAE-SDB 是判定逻辑，PCN 是落到工程现场的那个点。

没有 PCN，TPCA 作为方法论可以说得通，但现场还是不知道：在哪取信号、在哪判定、在哪输出、在哪记录。

有了 PCN，问题才能落位：这个节点设在哪、看哪个当前状态、判哪个目标状态、读哪些信号、怎么映射到 C/A/E、怎么从 S/D/B 判定、输出什么路径、记录什么内容。

这就是 PCN 成为 TPCA 最小工程单元的原因。

基础概念可参见：[Concepts｜核心概念](/zh/concepts/) 与 [TPCA / CAE-SDB 白皮书](/zh/whitepaper/)。

## 1. TPCA 是架构，PCN 是状态入口上的工程节点

TPCA 定义了一套处理方式：系统进入目标状态之前做前置判定。链条大致是：

```text
当前状态
→ 目标状态
→ 多源状态信号
→ C / A / E 状态映射
→ S / D / B 判定
→ CAE-SDB 判定结果
→ 多路径控制输出
→ 状态记录与追溯
```

链条说明的是“应该怎么处理”，但架构本身不能直接跑。现场还要明确：哪个位置需要判、哪些信号要纳入、哪个 ready 只是本机状态、哪个许可必须单独确认、哪个下游状态算执行链、哪个结果要进 HMI/MES/WCS/日志。

所以 TPCA 需要一个工程节点来落地。这个节点就是 PCN。

PCN 部署在目标状态入口前，读多源状态信号，做 C/A/E 映射和 S/D/B 判定，输出多路径控制结果，同时做状态记录。

CAE-SDB 是 PCN 内部用的判定逻辑：

C / A / E 说明问题来自哪个状态变量域：

C = Condition，条件状态；  
A = Authority，许可状态；  
E = Execution Chain，执行链状态。

S / D / B 说明该状态变量域的问题属于什么性质：

S = Structure，结构完整性；  
D = Dynamics，动态时序有效性；  
B = Boundary，控制边界。

举个例子：视觉结果是 C，安全许可是 A，下游可承接是 E。进一步看，视觉结果没接入是 C-S，结果过期是 C-D，安全许可没成立是 A-B，下游状态没刷新是 E-D。

CAE-SDB 把模糊的 NG 展开成结构化的判定。PCN 再明确：这个判定在哪个入口触发、读什么状态、转成什么路径、怎么记。

## 2. PCN 的最小闭环：位置、输入、判定、输出、记录

一个 PCN 至少需要明确以下内容。

| 项目 | 说明 |
|---|---|
| 当前状态 / 当前阶段 | 系统现在处于什么状态 |
| 目标状态 / 目标阶段 | 系统准备进入什么状态 |
| PCN 设置位置 | 判定节点设置在目标状态入口前的哪个位置 |
| 多源状态信号 | 来自设备、PLC、机器人、视觉、安全、MES、WCS、HMI 或其他系统的状态 |
| C / A / E 映射 | 哪些状态属于条件，哪些属于许可，哪些属于执行链 |
| S / D / B 判定 | 状态是否存在结构缺失、动态异常或控制边界问题 |
| 多路径控制输出 | 允许进入、等待、重识别、回流、异常分流、下游协调、禁止进入、安全锁定、人工确认等 |
| 状态记录 | 记录输入、判定结果、控制路径和追溯信息 |

这个表就是 PCN 的基本模板。

PCN 是最小工程单元，不是因为功能少，而是因为它包含了落地所需的最小闭环。

这个闭环包括五件事。

第一，有明确的目标状态入口。没有目标状态，就没有“进入前”的判定对象。等待→抓取、放置完成→压装、任务池→任务派发、车辆在线→可持续协同搬送，都可以设 PCN。

第二，有多源状态输入。现场的问题通常不只来自一个信号。机器人 ready、视觉 OK、安全许可、下游承接、上位放行、资源锁、工件状态、路径状态、结果回写，可能分布在不同的设备和系统里。

第三，有 C/A/E 状态映射。多源信号不能只列成一堆。PCN 需要判断它们在目标状态进入前起什么作用：条件、许可，还是执行链。

第四，有 S/D/B 判定。同样是条件状态，可能是结构缺失、动态不可信或控制边界。同样是许可状态，可能是来源未定义、许可撤销或触发禁止边界。同样是执行链状态，可能是下游未定义、反馈未刷新、需要协调/回流/隔离。

第五，有控制输出和状态记录。判定结果要转成控制路径，判定过程要形成记录。没输出，PCN 就是个诊断说明；没记录，PCN 没法复盘、改善、复用。

关于状态迁移条件为什么需要从隐含经验转成工程对象，可参见：[为什么状态迁移条件必须显式化？](/zh/notes/explicit-state-transition-conditions/)

## 3. PCN 与 interlock、报警、HMI / MES / WCS 的关系

interlock 在很多系统里已经存在了。它能阻止错误动作，回答“让不让”。

PCN 做的事更大一点——它关注目标状态进入前的整个判定过程，进一步回答：不让进是哪个域的问题、属于结构/动态/边界、有没有涉及关键许可、执行链能不能接、下一步是等、重识别、回流、协调、降级、禁止、安全锁定还是人工确认、这次判定怎么记、同类问题下次怎么复用。

PCN 可以读 interlock 的结果，也可以把 interlock 放进 A 许可或 C 条件里。

报警通常是异常发生后的提示。PCN 关注的是进入目标状态之前——抓取失败后的报警是动作后的结果，PCN 看的是进入抓取阶段之前，工件、视觉、安全、路径、夹爪、下游、异常分流、结果回写这些状态到底成不成。

PCN 也可以部署在不同层级：

在 PLC / HMI 场景中，PCN 可以对应自动化执行单元进入目标物理阶段前的判定节点。  
在 MES / WCS 场景中，PCN 可以对应任务派发、资源放行、站点承接、群体协同搬送恢复等目标执行路径前的判定节点。  
在生产 DX 场景中，PCN 可以对应人员、设备、系统、流程之间的状态迁移入口。

HMI、MES、WCS 可以显示 PCN 的判定结果。PLC、边缘控制器、WCS 或上位系统可以承接 PCN 的控制输出。日志系统和数据平台可以记录 PCN 的状态履历。

PCN 的形态可以随层级变化，但基本骨架不变：目标状态入口、多源状态输入、C/A/E 映射、S/D/B 判定、多路径控制输出、状态记录。

## 4. 一个例子：机器人进入抓取阶段前的 PCN

用视觉识别输送线机器人分拣单元来看。

当前状态：机器人等待，工件进入视觉识别区域，系统准备判断是否进入抓取阶段。

目标状态：机器人进入抓取阶段。

PCN 位置：机器人抓取动作启动之前。

多源状态信号：视觉识别结果、工件存在、工件位置和姿态、识别结果有效性、机器人自动模式、机器人 ready、路径可达、安全许可、主输送带状态、返回输送带状态、异常输送带状态、正常投放位可接收、上位系统放行。

这些信号可以映射为：

C 条件：工件存在、视觉结果有效、位置和姿态满足抓取要求；  
A 许可：安全许可、机器人自动模式、上位系统放行、区域许可；  
E 执行链：机器人路径可达、夹爪可用、正常投放位可承接、返回路径可用、异常分流路径可用、结果回写可用。

S / D / B 判定进一步看：这些状态有没有定义接入、是否有效同步可信、当前是等、重识别、回流、异常分流、禁止进入还是人工确认。

控制输出可能是：允许抓取、等待、重识别、重定位、放弃抓取并回流、异常品分流、禁止进入、安全锁定、人工确认、状态记录。

这个例子说明：PCN 不是在机器人失败后解释报警，而是在机器人进入抓取阶段前，把目标入口的结构化判定做完。

相关案例可参见：[自动化执行单元前置判定案例](/zh/cases/automation-execution-unit-pre-control/)

## 5. PCN 履历与 PCN 网络

PCN 不只输出当前路径，还生成履历。

PCN 履历记录：当前状态、目标状态、输入信号、C/A/E 映射结果、S/D/B 判定结果、控制路径、时间、追溯编号。

普通日志记“发生了什么”。PCN 履历进一步记：为什么进不了目标状态、哪个域不成立、问题属于什么性质、当时选了哪条路、同类问题是否反复出现。

这些履历用于复盘和改善：哪个入口经常失败、哪类许可经常没成立、哪类下游承接经常不同步、哪个执行链边界经常触发异常分流、哪些经验需要标准化、哪些信号需要补充定义。

单个 PCN 对应一个入口。多个 PCN 连起来就是状态迁移前置控制网络。

一条自动化产线里，PCN 可以分布在：工件进入识别前、机器人进入抓取前、机器人进入投放前、压装单元进入压装前、检测单元进入检测前、异常品进入分流前、AGV 进入站点前、缓存区释放前、MES/WCS 派发任务前、下游工位接收前。

多个 PCN 的履历积累后，能看出：哪类入口最容易失败、条件问题集中在哪、许可问题集中在哪、执行链问题集中在哪、哪些是结构缺失、哪些是动态不可信、哪些是边界不清。

这就是 PCN 网络的价值——把零散的状态迁移经验，变成可观察、可比较、可复盘的结构。

关于多个 PCN 如何连接成前置控制网络，可参见：[多个 PCN 如何形成状态迁移前置控制网络？](/zh/notes/pcn-network-structure/)

## 小结

TPCA 是架构，CAE-SDB 是判定逻辑，PCN 是落到工程现场的最小单元。

PCN 把方法论变成了现场可落位的东西：在哪判、判什么目标状态、读什么信号、怎么映射 C/A/E、怎么判定 S/D/B、输出什么路径、怎么记录和复盘。

没有 PCN，TPCA 停留在方法论层。

有了 PCN，TPCA 才能变成工程现场的一个个前置判定节点。

从单个 PCN 到多个 PCN 网络，TPCA/CAE-SDB 才能逐步形成制造系统中的状态迁移前置控制结构。
