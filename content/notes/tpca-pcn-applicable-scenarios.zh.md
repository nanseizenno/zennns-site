# TPCA / PCN 适用场景分析

TPCA / CAE-SDB 处理的问题是：技术系统进入目标状态前，如何基于多源状态信号完成前置判定，并输出控制路径。

PCN 是 TPCA 的最小工程单元。判断一个对象能不能纳入 PCN，不看它是设备、人、系统还是数据，而看它能不能进入下面这条工程链：


- 采集 / 记录
- → C / A / E 映射
- → S / D / B 判定
- → 多路径控制输出
- → 状态记录与追溯


能解释，不等于应该纳入。  
能纳入，必须能进入这条链。

---

## 1. 四个纳入条件

一个对象要纳入 PCN，至少同时满足四个条件。

| 层级 | 条件 | 判断重点 |
|---|---|---|
| 基础层 | 能被系统采集或记录 | 状态是否可读、可记录、可追溯 |
| 结构层 | 能映射到 C / A / E | 状态是否有明确变量域位置 |
| 分析层 | 能参与 S / D / B 判定 | 状态是否能分析结构、动态或边界性质 |
| 控制层 | 能影响多路径控制输出 | 状态变化是否能改变系统下一步动作 |

这四个条件缺一不可。

仅能采集，只是数据。  
能归类但不能判定，只是状态。  
能判定但不影响控制输出，只是记录。  
只有能改变控制路径，才是 PCN 的有效输入。

---

## 2. 排除标准

一个对象不应纳入 PCN，如果它属于以下情况：

- 不能被系统采集或记录；
- 只能依赖口头沟通、纸质文件、个人经验或事后询问；
- 无法映射到 C / A / E；
- 无法参与 S / D / B 判定；
- 不影响多路径控制输出；
- 只是背景信息、统计信息或组织管理信息；
- 主要问题来自部门协作、审批责任、会议判断、管理制度或组织结构。

注意，TPCA / CAE-SDB 不排除人，它排除的是无法系统化为技术判定要素的人类组织管理行为。

---

## 3. 人参与的生产 DX 场景如何判断

生产 DX、组装线、质量流转、设备复归、异常处置中，经常有人参与。关键不在于有没有人，而在于人的动作有没有被系统化。

人的动作如果满足以下条件，就可以纳入：

1. 能被系统采集或记录；
2. 能映射到 C / A / E；
3. 能参与 S / D / B 判定；
4. 能影响多路径控制输出。

| 人的动作 | 可纳入方式 |
|---|---|
| 操作员在 HMI 上确认工件已放置 | C：条件状态 |
| 操作员扫码确认工件 ID | C：对象识别条件 |
| 质检员录入首件 OK / NG | A：质量放行许可，或 C：质量条件 |
| 班组长释放生产批次 | A：上位许可 / 人工许可 |
| 保全员完成点检并解除待确认 | A：保全许可，或 E：执行链恢复 |
| 作业员选择回流、隔离、报废、再检 | B：边界状态下的人工确认路径 |
| 工程师确认异常复归 | A：复归许可，或 B：边界处置 |

这种生产相关的动作可以纳入。  
但是关于人的组织关系、会议判断、责任协调、审批层级，不应作为 PCN 主线对象。

---

## 4. 组装线例子

以组装线为例：

```text
当前状态：部品已到位，上一工序完成，设备等待确认
目标状态：进入下一组装工序、检测、包装或出货前检查
PCN 位置：目标工序进入前
```

此时，PCN 读取的信号可以包括：

| 信号来源 | 代表信号 |
|---|---|
| HMI | 操作员确认、人工放行、异常确认 |
| 扫码系统 | 工件 ID、部品 ID、作业者 ID、工序通行记录 |
| 质量系统 | 首件确认、抽检结果、NG 判定、质量放行 |
| MES | 工单状态、批次状态、工序许可 |
| 设备控制器 | 设备 ready、夹具状态、治具到位、传感器状态 |
| 安全系统 | 安全门、光栅、急停、区域许可 |
| 保全系统 | 点检完成、保全解除、复归确认 |
| 下游工位 | 可接收状态、缓存容量、包装准备状态 |

这些信号可以映射为：

| 变量域 | 代表项 |
|---|---|
| C：条件状态 | 工件 ID 正确、部品到位、装配完成、治具到位、参数匹配 |
| A：许可状态 | 质检放行、班组长放行、人工确认、保全解除、安全许可、MES 工序许可 |
| E：执行链状态 | 下游可接收、检测设备可用、包装路径可用、回流或隔离路径可用 |

控制输出可以是：

| 判定结果 | 控制路径 |
|---|---|
| C / A / E 均成立 | 允许进入下一工序 |
| 工件 ID 与工单不一致 | 禁止进入，要求重扫或人工确认 |
| 质检放行未成立 | 等待质检或禁止进入量产 |
| 人工确认与设备状态冲突 | 人工复核 |
| 下游工位不可接收 | 等待、缓存或下游协调 |
| NG 状态明确 | 隔离、报废或再检 |
| 保全解除未成立 | 禁止自动运行 |
| 多源状态冲突 | 安全锁定或人工确认 |

这个例子说明：人的动作可以进入 PCN，但前提是它已经成为系统可读、可判定、可控制的状态输入。

## 5. 为什么组织管理问题也能解释，但不应纳入

很多组织管理问题也可以用 C / A / E 做表面解释。

例如审批流程：

| C / A / E | 表面对应 |
|---|---|
| C：条件 | 资料是否齐全、申请内容是否明确、风险说明是否充分 |
| A：许可 | 审批人是否同意、权限是否满足、上级是否放行 |
| E：执行链 | 审批通过后是否有人执行、后续流程是否接续 |

会议决策、项目推进、采购审批、人事流程也可以做类似拆解。

这说明 C / A / E 有抽象解释力，但这类场景通常不能形成可靠的 PCN。

核心原因在于：牵涉到人类组织时，C 往往不是完整、客观、可公开、可系统采集的技术条件。

在技术系统中，C 可以来自传感器、PLC、MES、WCS、视觉系统、安全系统、质量系统或 HMI 记录。这些状态可以定义、接入、观测、记录和复盘。

但是在组织管理问题中，C 常常来自人的判断、部门立场、隐性资源、风险偏好、责任边界、优先级变化和未公开信息。

例如：

- 资料是否“足够充分”；
- 风险是否“可以接受”；
- 当前时机是否“适合推进”；
- 某部门是否“愿意配合”；
- 负责人是否“真实认可”；
- 资源是否“实际可用”；
- 上级是否“倾向支持”。

这些条件往往模糊、不稳定，也不一定会被完整公开。

因此，即使可以把它们写成 C，也很难得到完整、可靠、可复盘的 C。

C 不完整，后续的 A / E 判断也会失真。  
A 可能只是形式许可，不代表真实放行。  
E 可能看似有人承接，但实际执行仍取决于人的意愿、资源协调和组织关系。

TPCA / PCN 需要的是可观测状态，不是人的真实想法。

如果一个场景的关键条件隐藏在人际关系、部门利益、责任规避、风险态度或未公开信息中，即使存在“当前状态 → 目标状态”，也不适合作为 PCN 的主线对象。

能解释，说明 C / A / E 有抽象能力。  
不纳入，说明 PCN 有工程边界。

## 6. 适合与不适合的场景

| 适合纳入 | 原因 |
|---|---|
| 自动化执行单元不能进入下一阶段 | 当前阶段、目标阶段、信号和控制路径清晰 |
| AGV / AMR 群控停滞 | 群体状态可观测，任务、许可、执行链可拆解 |
| MES / WCS 只能记录但不能解释停滞 | 协同状态可以结构化诊断 |
| 机器人抓取、压装、检测、交接前置判定 | 目标物理执行阶段清晰 |
| 组装线人工确认与质检放行 | 人工动作可系统记录、映射、判定并影响控制路径 |
| 生产 DX 中的质量流转 | 质量状态、许可和异常路径可追溯 |
| 设备复归前确认 | 保全、安全、复归和执行链状态可拆解 |
| AI / 工具调用前置控制 | 调用条件、授权、执行就绪状态可记录，控制路径明确 |

| 不适合作为主线 | 原因 |
|---|---|
| 普通审批流程 | C 条件模糊，核心是组织流程 |
| 会议决策流程 | 条件和许可依赖人的判断，工程控制输出弱 |
| 人事、采购、行政流程 | 容易滑向 BPM 或管理制度优化 |
| 普通项目管理流程 | 多数问题来自组织协作和责任分配 |
| 组织架构设计 | 无法形成稳定 C / A / E，也不产生直接控制路径 |
| 单纯统计报表 | 只能观察，不能参与判定和控制 |
| 纯趋势监控数据 | 如果不影响目标状态进入，不属于 PCN 核心输入 |
| CI / CD 发布管理 | 主要矛盾偏 DevOps 发布治理、组织协同和管理策略 |

不适合场景不是完全不能解释。  
问题在于：即使用 TPCA / CAE-SDB 解释，也难以体现其核心技术价值。

---

## 7. 小结

TPCA / PCN 的适用场景，不能只按行业、对象名称或是否有人参与来判断。

真正的判断标准是：


- 能否被系统采集或记录；
- 能否映射到 C / A / E；
- 能否参与 S / D / B 判定；
- 能否影响多路径控制输出。

四个条件全部满足，才有纳入基础。

TPCA / CAE-SDB 不排除人，它排除的是无法系统化为技术判定要素的人类组织管理行为。

组织管理问题之所以不适合作为主线，不是因为 C / A / E 解释不了，而是因为 C 往往模糊、隐性、不完整、不可系统采集。C 不可靠，后续 A / E 和 S / D / B 判定也无法形成可靠工程链。

能解释，不等于应该纳入。  
能纳入，必须进入：

```text
采集 / 记录
→ C / A / E 映射
→ S / D / B 判定
→ 多路径控制输出
→ 状态记录与追溯
```
