## 状态机、安全互锁、报警诊断、AI 诊断、调度控制与前置判定层的边界说明

## 1. 本文目的

本文用于说明 TPCA / CAE-SDB 状态迁移前置控制架构与既有工业自动化体系之间的边界关系。

工业自动化现场已经存在成熟的状态控制、安全控制、报警管理、故障诊断、调度控制、数据分析和运行验证方法。TPCA / CAE-SDB 不替代这些体系，也不试图重新定义它们。

TPCA / CAE-SDB 关注的是另一个具体问题：

当系统准备进入目标状态、目标执行路径或目标物理执行阶段之前，如何判断进入条件是否成立、许可是否成立、执行链是否可接续；相关信号是否完整、可信、处于可控边界；以及不成立时应走哪条控制路径。

本文只讨论边界，不讨论优劣比较。

---

## 2. TPCA / CAE-SDB 的处理对象

TPCA / CAE-SDB 的处理对象，不是一般意义上的“异常”，也不是所有工业控制问题。

它处理的是目标状态进入前的判断问题。

典型现场表现包括：

- ready 了，但系统不能进入下一动作；
- interlock NG，但不能直接判断是哪类前置条件不成立；
- alarm 已经出现，但不能说明它对当前目标状态进入的影响；
- waiting / pending / blocked 持续存在，但无法判断等待来自条件、许可还是执行链；
- 调度结果已经生成，但任务不能进入有效执行；
- 设备局部状态正常，但进入下一阶段后执行链无法接续。

TPCA / CAE-SDB 的基本处理链条为：

当前状态 / 当前阶段  
→ 目标状态 / 目标阶段  
→ 多源状态信号  
→ C/A/E 状态映射  
→ S/D/B 判定  
→ CAE-SDB 判定结果  
→ 多路径控制输出  
→ 状态记录与履历

其中：

C = Condition，条件状态。  
表示进入目标状态前，现场条件、对象条件、识别条件、数据条件、任务条件、参数条件等是否成立。

A = Authority，许可状态。  
表示系统是否允许进入目标状态，包括安全许可、区域许可、上位许可、人工确认、权限、授权、资源锁、对方设备许可等。

E = Execution Chain，执行链状态。  
表示进入目标状态后，执行链是否能够接续。E 不等于单体 ready。E 包括本体设备、末端执行机构、下游承接、替代路径、回退路径、异常排出路径、结果上传或回写链路等。

S = Structure，结构完整性。  
判断所需信号、接口、映射关系、许可来源、路径、角色、执行链边界是否已经定义、接入并可观测。

D = Dynamics，动态时序有效性。  
判断信号是否超时、未刷新、抖动、冲突、延迟、不同步、低置信度、许可撤销或处于切换中。

B = Boundary，控制边界。  
判断系统处于未完全确定或边界状态时，下一步应等待、重试、重识别、回流、降级、禁止、人工确认、安全锁定还是异常隔离。

TPCA / CAE-SDB 的输出不是原因标签，而是控制路径，例如允许进入、等待、重识别、重采样、重定位、回流、异常分流、下游协调、资源释放、降级执行、禁止进入、安全锁定、人工确认、异常隔离和增强记录。

## 3. 与既有工业自动化体系的边界

### 3.1 与状态机、SFC 和顺序控制的边界

状态机、SFC 和 PLC 顺序控制负责定义状态、步骤、动作顺序和状态迁移关系。

它们回答：

系统有哪些状态；  
状态之间如何切换；  
满足什么条件时进入下一状态。

TPCA / CAE-SDB 不替代状态机，也不替代 SFC。

TPCA / CAE-SDB 部署在目标状态进入前，用于展开状态迁移条件本身。它判断迁移条件中的信号是否定义、是否有效、许可是否成立、执行链是否能够接续，以及不满足时应进入哪条控制路径。

边界可以概括为：

状态机负责流程推进。  
PCN 负责迁移前判定。

---

### 3.2 与互锁和安全控制的边界

安全 PLC、安全继电器、安全回路、急停、光栅、安全门和区域进入许可，负责现场安全功能。

TPCA / CAE-SDB 不替代安全系统，不绕过安全系统，也不降低安全系统的优先级。

TPCA / CAE-SDB 接收安全许可状态，并将其作为 A 许可状态参与判定。

当安全门未关闭、光栅被触发、急停被按下、区域许可未成立或上位系统禁止放行时，即使 C 条件成立、E 执行链可接续，也不得允许进入目标状态。

边界可以概括为：

安全系统负责安全保障。  
PCN 负责将安全许可纳入目标状态进入前的结构化判定。

---

### 3.3 与报警管理和故障诊断的边界

报警管理负责异常检测、报警通知、报警优先级、报警记录和报警生命周期管理。

故障诊断负责识别故障现象、故障模式、故障原因或故障传播关系。

TPCA / CAE-SDB 不替代报警系统，也不替代故障诊断方法。

TPCA / CAE-SDB 处理的是报警或故障信息对“当前目标状态是否可以进入”的影响。

例如：

机器人报警，是目标抓取阶段不能进入的前置阻断，还是进入后失败的结果；  
视觉报警，影响的是 C 条件，还是 C-D 动态有效性；  
下游报警，是 E 执行链不可接续，还是 A 许可未成立；  
通信报警，是接口结构未接入，还是运行中动态超时。

TPCA / CAE-SDB 将报警和故障信息放入 C/A/E 与 S/D/B 结构中，判断其变量域、判定性质和控制路径。

边界可以概括为：

报警系统负责提示异常。  
故障诊断负责分析故障。  
PCN 负责判断异常或故障信息对当前目标状态进入的影响。

---

### 3.4 与 AI 诊断和数据分析的边界

AI 诊断和数据分析负责模式识别、趋势分析、异常分类、预测性维护和辅助改善建议。

TPCA / CAE-SDB 不依赖 AI 才能成立。AI 也不直接替代 PCN 的现场判定。

PCN 运行时可以记录当前状态、目标状态、多源状态信号、C/A/E 映射结果、S/D/B 判定结果、多路径控制输出、时间戳、TraceLogID 和人工确认结果。

这些记录可以为后续数据分析或 AI 辅助复盘提供结构化输入。

但现场控制逻辑、PCN 配置修改、控制边界变更、试行和回退，仍应由工程师、现场负责人和既有管理流程决定。

边界可以概括为：

AI 负责辅助分析。  
PCN 负责运行时前置判定和履历记录。

---

### 3.5 与 MES / WCS 调度和协同控制的边界

MES / WCS / AGV / AMR 调度系统负责生产任务、资源、路径、站点、车辆和优先级之间的匹配。

它们回答：

谁执行任务；  
何时执行；  
走哪条路径；  
使用哪个资源；  
如何避免冲突；  
如何提高吞吐。

TPCA / CAE-SDB 不替代调度算法，也不替代路径规划、交通管制或资源优化。

TPCA / CAE-SDB 判断的是：调度结果进入执行前，条件、许可和执行链是否成立。

例如：

任务已经存在，但车辆未执行；  
车辆空闲，但任务没有进入有效执行；  
多台车处于 waiting、blocked 或 pending；  
资源锁未释放，后续任务无法进入；  
站点不可接收，任务滞留；  
车辆集中充电，群体执行能力下降。

这些问题不一定是调度算法本身错误，也可能是目标执行状态进入前的条件、许可或执行链没有成立。

边界可以概括为：

调度系统负责生成执行方案。  
PCN 负责判定执行方案是否具备进入执行路径的前置条件。

---

### 3.6 与形式化验证、模型检查和运行时验证的边界

形式化验证和模型检查通常用于验证模型是否满足指定性质，重点在设计阶段、模型层和规格验证。

运行时验证关注系统运行过程中事件轨迹是否违反既定规格。

TPCA / CAE-SDB 与这些方向在“状态、迁移、约束、属性判断”上存在相邻关系，但工程位置不同。

TPCA / CAE-SDB 面向工业现场中的目标状态入口：

- 不要求先建立完整形式模型；
- 从现场可获得的多源状态信号出发；
- 将信号映射为 C/A/E 状态变量；
- 从 S/D/B 判定结构完整性、动态可信性和控制边界；
- 输出等待、重识别、回流、协调、降级、禁止进入、安全锁定、人工确认等工程控制路径；
- 记录判定履历，用于复盘、改善和再部署。

边界可以概括为：

形式化验证关注模型是否满足性质。  
运行时验证关注执行过程是否违反规格。  
PCN 关注目标状态进入前，现场系统是否具备进入条件，以及不具备时应走哪条控制路径。

## 4. 与现场压缩状态的关系

工业现场中，很多复杂状态会被压缩成简单显示。

常见压缩状态包括：

- ready / not ready；
- OK / NG；
- true / false；
- interlock OK / interlock NG；
- alarm；
- waiting；
- pending；
- blocked；
- idle；
- dispatch failed；
- handshake timeout。

这些状态在现场有用，但它们通常只是压缩结果。

ready 只能说明某个设备、机构或模块处于可运行状态，不能说明进入目标阶段后整个执行链一定可以接续。

interlock NG 可以说明某个互锁条件未满足，但不能直接说明它属于 C 条件、A 许可还是 E 执行链问题。

alarm 可以说明异常发生，但不能直接说明该异常对当前目标状态进入有什么影响。

waiting 可以说明系统等待中，但不能说明等待来自条件不成立、许可未成立、执行链不可接续，还是动态信号未刷新。

pending 可以说明任务未进入执行，但不能说明任务条件、调度许可、资源锁、路径、车辆状态、下游承接中哪一部分未成立。

TPCA / CAE-SDB 的作用，是把这些压缩状态展开为：

- 属于 C、A、E 哪个变量域；
- 属于 S、D、B 哪类判定性质；
- 是否影响当前目标状态进入；
- 应进入哪条控制路径；
- 是否需要记录履历用于复盘。

因此，ready、alarm、waiting、pending 等不是 TPCA / CAE-SDB 的替代对象，而是 TPCA / CAE-SDB 需要展开和解释的现场压缩状态。

---

## 5. 结论

TPCA / CAE-SDB 的边界可以归纳如下：

状态机和 SFC 负责定义状态与迁移。  
安全控制负责危险动作禁止和安全功能实现。  
报警管理负责异常检测、通知和记录。  
故障诊断负责故障现象、模式和原因分析。  
AI 诊断和数据分析负责复杂模式识别和辅助改善。  
MES / WCS 调度负责任务、资源、路径和主体分配。  
形式化验证和运行时验证负责模型或运行过程的性质检查。

TPCA / CAE-SDB 处理的是目标状态进入前的运行时前置判定与多路径控制。

它将复杂工程系统中分散存在的状态信号、许可约束、执行链状态、报警、调度结果和人工确认，整理为 PCN 前置控制节点中的 C/A/E 状态映射、S/D/B 判定和多路径控制输出。

其工程定位可以概括为：

复杂工程系统目标状态进入前的运行时前置判定与多路径控制层。

它的价值在于，把原本分散在 PLC 程序、HMI 画面、MES / WCS 日志、机器人控制器、安全许可、调度系统和人工经验中的状态迁移前判断，转化为可配置、可显示、可追溯、可复用的 PCN 判定结构。
