## 状态机、安全互锁、报警诊断、AI 诊断、调度控制与前置判定层的边界说明

## 1. 本文目的

本文用于说明 TPCA / CAE-SDB 状态迁移前置控制架构在既有工业自动化理论版图中的位置。

TPCA / CAE-SDB 与状态机、SFC、安全互锁、报警管理、故障诊断、AI 诊断、MES / WCS 调度、形式化验证之间是什么关系？

工业自动化领域已经有成熟的状态控制、安全控制、报警管理、调度优化和故障诊断体系。TPCA / CAE-SDB 的位置，应放在这些既有体系之间理解。

它处理的核心问题是：

系统准备进入目标状态、目标执行路径或目标物理执行阶段之前，如何结构化判断条件是否成立、许可是否成立、执行链是否可接续；相关信号是否完整、可信、处于可控边界；以及下一步应走哪条控制路径。

---

## 2. 工业自动化中的主要参照系

### 2.1 状态机与顺序控制

状态机和顺序控制是工业自动化中最基础的控制表达方式。

代表形式包括：

- 有限状态机，FSM；
- IEC 61131-3 中的 SFC，Sequential Function Chart；
- UML 状态图；
- PLC 程序中的步进流程、模式切换和阶段控制逻辑。

其核心逻辑是：定义有限个状态，定义状态之间的迁移关系，并在满足迁移条件时，使系统从当前状态进入下一个状态。

在自动化设备中，常见表达包括：

- 等待阶段 → 抓取阶段；
- 抓取完成 → 搬送阶段；
- 放置完成 → 压装阶段；
- 检测完成 → 分流阶段；
- 搬送待机 → 交接阶段。

状态机处理的是流程控制、顺序控制、模式切换和状态推进问题。它回答的是：

系统有哪些状态；  
状态之间如何切换；  
满足什么条件时进入下一状态。

TPCA / CAE-SDB 与状态机的关系是：

状态机定义状态和迁移关系。  
TPCA / CAE-SDB 部署在目标状态进入前，对迁移前的条件、许可和执行链进行结构化判定。

状态机中的迁移条件通常会被写成一个 transition condition。  
TPCA / CAE-SDB 进一步展开这个 transition condition：

- 条件信号是否定义并接入；
- 条件信号是否仍然有效；
- 许可来源是否明确；
- 安全、区域、上位、人工确认等许可是否成立；
- 进入目标阶段后执行链是否能够接续；
- 当前状态是否处于边界；
- 不满足时应等待、重识别、回流、降级、禁止进入，还是人工确认。

因此，TPCA / CAE-SDB 可以理解为状态迁移入口前的工程判定层。

### 2.2 互锁与安全控制

互锁和安全控制是工业设备中最重要的控制边界之一。

代表体系包括：

- IEC 61508 功能安全；
- ISO 13849 机械安全相关控制系统；
- 安全继电器；
- 安全 PLC；
- 安全门、光栅、急停、区域进入许可等安全信号。

其核心逻辑是：通过硬件或软件控制机制，防止危险动作发生。安全信号通常具有最高优先级，必要时直接切断动力或禁止设备进入危险动作。

安全控制处理的是：

- 人员安全；
- 设备保护；
- 危险动作禁止；
- 安全功能是否满足要求；
- 安全回路是否成立；
- 风险是否处于可接受范围内。

TPCA / CAE-SDB 与安全控制的关系是：

安全系统承担安全功能本体。  
TPCA / CAE-SDB 接收安全许可状态，并将其映射到 A，也就是 Authority 许可状态。

在 TPCA / CAE-SDB 中，关键 A 可以构成独立必要约束。

当安全门未关闭、光栅被触发、急停被按下、区域许可未成立、上位系统禁止放行时，即使 C 条件成立、E 执行链看似可接续，也不得允许进入目标状态。

这里的边界必须清楚：

- 安全 PLC、安全继电器、安全回路仍负责安全功能；
- PCN 接收安全许可状态；
- CAE-SDB 将安全许可作为 A 域进行判定；
- A 不成立时，输出禁止进入、安全锁定、等待许可、人工确认等控制路径；
- PCN 不绕过现场安全系统，也不替代硬件安全回路。

TPCA / CAE-SDB 的作用，是在控制逻辑和状态迁移判定层，把安全许可纳入目标状态进入前的结构化判定。

### 2.3 报警管理与故障诊断

报警管理和故障诊断是工业现场长期使用的异常处理体系。

代表方法包括：

- ISA-18.2 报警管理；
- FTA，Fault Tree Analysis；
- FMEA，Failure Mode and Effects Analysis；
- 基于模型的故障诊断；
- 基于信号的故障诊断；
- 基于数据的故障诊断；
- 设备报警、过程报警、通信报警和系统报警。

报警与故障诊断的基本逻辑是：

检测异常信号；  
触发报警；  
提示操作员；  
记录事件；  
在部分系统中进一步进行原因分析和处置支持。

报警系统处理的是异常检测、报警通知、报警优先级、报警生命周期、报警洪泛治理、事件记录和事后追溯等问题。

但在复杂自动化系统中，报警本身经常不能直接回答状态迁移问题。

例如：

- 机器人报警，是目标抓取阶段不能进入的源头，还是进入后失败的结果；
- 视觉报警，影响的是 C 条件，还是 C-D 动态有效性；
- 下游报警，是 E 执行链不可接续，还是 A 许可未成立；
- 安全报警，应作为 A 域禁止进入，还是进入安全锁定路径；
- 通信报警，是结构未接入、动态超时，还是边界等待。

TPCA / CAE-SDB 对报警的处理方式，是将报警重新放入目标状态进入前的判定结构中。

它判断：

- 报警影响 C、A、E 中哪个变量域；
- 报警属于 S、D、B 中哪类判定性质；
- 报警是否直接影响当前目标状态进入；
- 报警应触发等待、重采样、重识别、下游协调、禁止进入、安全锁定，还是人工确认；
- 报警是否需要进入 PCN 履历，用于后续复盘。

因此，TPCA / CAE-SDB 可以把“报警显示”进一步转化为“状态迁移前置判定”。

这对现场有实际意义。很多设备并非缺少报警，而是报警与当前目标状态之间的关系没有被结构化表达。TPCA / CAE-SDB 处理的正是这部分关系。

### 2.4 数据驱动与 AI 诊断

数据驱动诊断和 AI 诊断是当前工业系统中非常活跃的方向。

代表方法包括：

- 机器学习分类；
- 深度学习故障诊断；
- 异常检测；
- 预测性维护；
- 数字孪生；
- 数据驱动根因分析；
- 大模型辅助运维分析。

其基本逻辑是：

采集大量历史数据；  
整理运行状态、传感器、报警和日志；  
训练或调用模型；  
识别异常模式；  
输出诊断结果、预测结果或改善建议。

数据驱动与 AI 诊断擅长发现复杂模式，尤其适合人工难以直接观察的高维数据关系。

但工业 AI 诊断在落地时经常遇到两个问题：

第一，原始数据多，但工程语义不够清楚。  
第二，模型能识别异常模式，但现场仍需要判断这个模式对当前控制路径意味着什么。

TPCA / CAE-SDB 与 AI 诊断的关系是：

TPCA / CAE-SDB 不依赖 AI 才能成立。  
AI 也不直接替代 PCN 的现场判定。  
PCN 可以为 AI 提供更清晰的结构化履历数据。

PCN 运行时可以记录：

- 当前状态；
- 目标状态；
- 多源状态信号；
- C/A/E 映射结果；
- S/D/B 判定结果；
- 多路径控制输出；
- 时间戳；
- TraceLogID；
- 人工确认结果；
- 后续是否复发。

这些数据比单纯原始日志更适合后续分析。

AI 可以在 PCN 履历积累之后，用于：

- 跨节点模式分析；
- 高频迁移失败节点识别；
- 候选改善方案生成；
- 效果预测；
- 复盘报告生成；
- 维护与调试知识沉淀。

但现场控制逻辑、PCN 配置修改、控制边界变更、试行和回退，仍应由工程师、现场负责人和既有管理流程决定。

TPCA / CAE-SDB 为 AI 提供结构化输入；AI 为 TPCA / CAE-SDB 的复盘和改善提供辅助分析。

### 2.5 调度与协同控制

MES / WCS / AGV / AMR 调度系统关注任务、资源、路径和执行主体之间的匹配关系。

代表领域包括：

- MES 任务管理；
- WCS 调度；
- AGV / AMR 路径规划；
- 交通管制；
- 资源锁管理；
- 站点分配；
- 缓存区控制；
- 任务优先级；
- Petri Net；
- 多主体协同控制。

其核心逻辑是：

根据任务、资源、约束条件和系统状态，生成调度方案，并将任务分配给执行主体。

调度与协同控制处理的是：

- 谁执行任务；
- 何时执行；
- 走哪条路径；
- 使用哪个资源；
- 如何避免冲突；
- 如何提高吞吐；
- 如何进行路径规划和资源优化。

但制造现场经常出现一种问题：

调度结果已经生成，任务也已经存在，但系统仍无法进入持续执行状态。

例如：

- MES 有任务，WCS 有记录，但车辆未执行；
- 车辆空闲，但任务没有被有效派发；
- 多台车处于 waiting、blocked、pending；
- 站点不可接收，导致任务滞留；
- 窄通道、自动门、电梯或交接区形成资源竞争；
- 资源锁未释放，后续任务无法进入；
- 车辆集中充电，导致可执行车辆比例下降；
- 下游工位缺料，但补料任务没有进入有效执行。

调度系统回答的是：任务如何分配、路径如何规划、资源如何配置。

TPCA / CAE-SDB 进一步回答：

调度结果进入执行前，是否具备条件、许可和执行链。

在 case1 中，WCS / 群控 PCN 可以读取 MES、WCS、AGV / AMR、站点、缓存区、路径资源、充电区、安全系统和下游工位等多源状态信号，计算群体未执行比例、群体辅助执行比例、群体能源补给比例、授权未成立比例等指标。

这些指标用于识别协同停滞结构，例如：

- 分配约束型；
- 资源竞争型；
- 能源偏在型；
- 授权未成立型；
- 未确定型。

进一步可以触发：

- 重新分配任务；
- 释放资源锁；
- 调整优先级；
- 路径复核；
- 通行权协调；
- 限流；
- 暂停低优先级任务；
- 限制同时充电；
- 改派缓存区；
- 增强记录；
- 人工确认；
- 异常隔离。

因此，TPCA / CAE-SDB 在 MES / WCS 中的定位，是协同停滞的前置判定和结构诊断层。

它使系统不只记录任务和状态，还能够进一步解释协同停滞结构。

## 3. TPCA / CAE-SDB 的边界

TPCA / CAE-SDB 与上述理论体系并行存在。它需要与既有系统协同使用。

### 3.1 不替代状态机

状态机和 SFC 仍负责定义流程、步骤、状态、迁移关系和动作顺序。

TPCA / CAE-SDB 部署在目标状态进入前，用于判定迁移条件本身是否完整、可信、处于可控边界。

状态机负责流程推进。  
PCN 负责迁移前判定。

### 3.2 不替代安全系统

安全 PLC、安全继电器、安全回路和功能安全设计仍负责现场安全功能。

TPCA / CAE-SDB 接收安全许可、区域许可、急停、光栅、安全门等信号，并将其作为 A 许可状态参与判定。

安全系统负责安全保障。  
PCN 负责将安全许可纳入状态迁移前置判定。

### 3.3 不替代报警系统

报警系统仍负责异常检测、报警通知、报警优先级、报警记录和报警生命周期管理。

TPCA / CAE-SDB 将报警信号重新放入 C/A/E 与 S/D/B 结构中，判断该报警对当前目标状态进入的影响。

报警系统负责提示异常。  
PCN 负责判断异常与当前状态迁移之间的关系。

### 3.4 不替代 AI 诊断

AI 诊断仍负责复杂模式识别、趋势分析、异常分类和辅助改善建议。

TPCA / CAE-SDB 提供结构化 PCN 履历数据，使 AI 的输入不再只是原始信号和日志，而是带有状态迁移语义的数据。

AI 负责模式分析。  
PCN 负责运行时前置判定和履历记录。

### 3.5 不替代调度算法

MES / WCS / AGV 调度系统仍负责任务分配、路径规划、交通管制和资源优化。

TPCA / CAE-SDB 判断调度结果进入执行前，条件、许可和执行链是否成立。

调度系统负责生成执行方案。  
PCN 负责判定执行方案是否具备进入执行路径的条件。

## 4. TPCA / CAE-SDB 补充什么

TPCA / CAE-SDB 补充的是目标状态进入前的结构化前置判定层。

其基本处理链条为：

当前状态 / 当前阶段  
→ 目标状态 / 目标阶段  
→ 多源状态信号  
→ C/A/E 状态映射  
→ S/D/B 判定  
→ CAE-SDB 判定结果  
→ 多路径控制输出  
→ 状态记录与履历  
→ 复盘、改善与再部署

### 4.1 C/A/E 状态映射

C/A/E 表示目标状态进入前的三个状态变量域。

C = Condition，条件状态。  
表示进入目标状态前，现场条件、对象条件、识别条件、数据条件、任务条件、参数条件等是否成立。

A = Authority，许可状态。  
表示系统是否允许进入目标状态，包括安全许可、区域许可、上位许可、人工确认、权限、授权、资源锁、对方设备许可等。

E = Execution Chain，执行链状态。  
表示进入目标状态后，执行链是否能够接续。E 不等于单体 ready。E 包括本体设备、末端执行机构、下游承接、替代路径、回退路径、异常排出路径、结果上传或回写链路等。

C/A/E 的作用，是把原本混在一起的 ready、OK、NG、alarm、waiting、pending 等状态，拆成不同的状态变量域。

### 4.2 S/D/B 判定性质

S/D/B 表示对 C/A/E 状态变量进行判定时使用的三类性质。

S = Structure，结构完整性。  
判断所需信号、接口、映射关系、许可来源、路径、角色、执行链边界是否已经定义、接入并可观测。

D = Dynamics，动态时序有效性。  
判断信号是否超时、未刷新、抖动、冲突、延迟、不同步、低置信度、许可撤销或处于切换中。

B = Boundary，控制边界。  
判断系统处于未完全确定或边界状态时，下一步应等待、重试、重识别、回流、降级、禁止、人工确认、安全锁定还是异常隔离。

S/D/B 不是设备分类，也不是异常标签。  
S/D/B 是 PCN 对 C/A/E 状态变量进行分析时使用的判定性质。

### 4.3 多路径控制输出

TPCA / CAE-SDB 的最终输出不是单纯原因标签，也不是报警显示，而是控制路径。

典型输出包括：

- 允许进入；
- 等待；
- 重识别；
- 重采样；
- 重定位；
- 回流；
- 异常分流；
- 下游协调；
- 资源释放；
- 降级执行；
- 备用路径；
- 禁止进入；
- 安全锁定；
- 人工确认；
- 异常隔离；
- 增强记录。

不同变量域和不同判定性质，对应不同控制路径。

C 条件不成立时，可能输出重识别、重采样、重定位、等待或人工确认。  
A 许可不成立时，可能输出禁止进入、安全锁定、等待许可刷新或人工确认。  
E 执行链不成立时，可能输出等待、下游协调、回流、降级执行、备用路径或异常隔离。  
S 结构缺失时，通常需要补充信号、接口、映射关系或许可来源。  
D 动态异常时，通常需要等待稳定、重采样、重新同步或保存状态。  
B 控制边界触发时，需要在等待、重试、降级、禁止、安全锁定、人工确认之间选择控制路径。

### 4.4 PCN 履历记录

PCN 不只完成一次判定，还应记录判定过程。

典型记录内容包括：

- 当前状态；
- 目标状态；
- 多源状态信号；
- 信号时间戳；
- C/A/E 映射结果；
- S/D/B 判定结果；
- 多路径控制输出；
- 人工确认结果；
- TraceLogID；
- 后续处理结果。

多个 PCN 的履历可以形成状态迁移前置判定网络。

其价值包括：

- 识别高频迁移失败节点；
- 记录控制路径触发历史；
- 分析条件、许可、执行链之间的薄弱点；
- 支撑现场复盘；
- 支撑新人培训；
- 支撑模板复用；
- 支撑 AI 辅助分析和改善方案生成。

## 5. 与形式化验证、模型检查和运行时验证的相邻关系

从理论问题形式看，TPCA / CAE-SDB 与形式化验证、模型检查和运行时验证存在相邻关系。

这些方向都关注：

- 系统状态；
- 状态迁移；
- 约束条件；
- 属性是否满足；
- 安全性、活性、可达性或一致性；
- 系统是否可以进入某种状态。

但 TPCA / CAE-SDB 的工程位置不同。

形式化验证和模型检查通常偏向：

- 设计阶段；
- 模型层；
- 规格验证；
- 离线分析；
- 证明某个模型是否满足指定性质。

运行时验证更接近 TPCA / CAE-SDB，因为它关注系统运行过程中的轨迹、事件和性质满足情况。

但是，TPCA / CAE-SDB 的重点仍然不同。

TPCA / CAE-SDB 面向工业现场中的目标状态入口：

- 不要求先建立完整形式模型；
- 从现场可获得的多源状态信号出发；
- 将信号映射为 C/A/E 状态变量；
- 从 S/D/B 判定结构完整性、动态可信性和控制边界；
- 输出等待、重识别、回流、协调、降级、禁止进入、安全锁定、人工确认等工程控制路径；
- 记录判定履历，用于复盘、改善和再部署。

因此，可以这样理解：

形式化验证更关注模型是否满足性质。  
运行时验证更关注执行过程是否违反规格。  
TPCA / CAE-SDB 更关注目标状态进入前，现场系统是否具备进入条件，以及不具备时应走哪条控制路径。

三者在状态、迁移和属性判断上相邻，但工程目标、输入对象和输出形式不同。

## 6. 与现场压缩状态的关系

工业现场中，很多复杂状态会被压缩成简单显示。

常见压缩状态包括：

- ready / not ready；
- OK / NG；
- true / false；
- interlock OK / interlock NG；
- alarm；
- waiting；
- pending；
- blocked；
- idle；
- dispatch failed；
- handshake timeout。

这些状态在现场有用，但它们通常只是压缩结果。

例如：

ready 只能说明某个设备、机构或模块处于可运行状态，但不能说明进入目标阶段后，整个执行链一定可以接续。

interlock NG 可以说明某个互锁条件未满足，但不能直接说明它属于 C 条件、A 许可还是 E 执行链问题。

alarm 可以说明异常发生，但不能直接说明该异常对当前目标状态进入有什么影响。

waiting 可以说明系统等待中，但不能说明等待来自条件不成立、许可未成立、执行链不可接续，还是动态信号未刷新。

pending 可以说明任务未进入执行，但不能说明任务条件、调度许可、资源锁、路径、车辆状态、下游承接中哪一部分未成立。

TPCA / CAE-SDB 的作用，是把这些压缩状态展开。

展开方式为：

ready / not ready  
→ 判断其属于 C、A、E 哪个变量域，是否仅为局部 ready，是否覆盖执行链接续。

interlock NG  
→ 判断互锁对应条件、许可还是执行链，是否属于结构缺失、动态异常或控制边界。

alarm  
→ 判断报警与当前目标状态进入的关系，确定其影响变量域和控制路径。

waiting / pending / blocked  
→ 判断停滞来自条件、许可、执行链，还是结构、动态、边界问题。

dispatch failed  
→ 判断调度结果进入执行前，任务条件、授权许可、资源锁、执行主体、路径、下游承接是否成立。

因此，ready、alarm、waiting 等不是 TPCA / CAE-SDB 的理论参照系，而是 TPCA / CAE-SDB 需要展开和解释的现场压缩状态。

## 7. 结论

TPCA / CAE-SDB 应放在既有工业自动化理论体系的边界处理解。

状态机和 SFC 负责定义状态与迁移。  
安全控制负责危险动作禁止和安全功能实现。  
报警管理和故障诊断负责异常检测、通知、记录和原因分析。  
数据驱动与 AI 诊断负责复杂模式识别和辅助改善。  
MES / WCS 调度负责任务、资源、路径和主体分配。  
形式化验证和运行时验证负责模型或运行过程的性质检查。

TPCA / CAE-SDB 处理的是目标状态进入前的运行时前置判定与多路径控制。

它将复杂工程系统中分散存在的状态信号、许可约束、执行链状态、报警、调度结果和人工确认，整理为 PCN 前置控制节点中的 C/A/E 状态映射、S/D/B 判定和多路径控制输出。

其工程定位可以概括为：

复杂工程系统目标状态进入前的运行时前置判定与多路径控制层。

它的价值在于，把原本分散在 PLC 程序、HMI 画面、MES / WCS 日志、机器人控制器、安全许可、调度系统和人工经验中的状态迁移前判断，转化为可配置、可显示、可追溯、可复用的 PCN 判定结构。
