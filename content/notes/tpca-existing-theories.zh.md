## 状态机、安全互锁、报警诊断、AI 诊断、调度控制与前置判定层的边界说明

## 1. 本文目的

本文用于说明 TPCA / CAE-SDB 状态迁移前置控制架构在既有工业自动化理论版图中的位置。

工业自动化领域已经有成熟的状态控制、安全控制、报警管理、调度优化、故障诊断和运行验证体系。TPCA / CAE-SDB 应放在这些既有体系之间理解。

它处理的核心问题是：

系统准备进入目标状态、目标执行路径或目标物理执行阶段之前，如何结构化判断条件是否成立、许可是否成立、执行链是否可接续；相关信号是否完整、可信、处于可控边界；以及下一步应走哪条控制路径。

本文重点说明 TPCA / CAE-SDB 与既有方法之间的边界、接口和补充位置。

---

## 2. 与工程结构化方法论的横向关系

TPCA / CAE-SDB 更适合放在“工程问题结构化方法”的谱系中理解。

成熟工程方法论的共同特征，是把原本依赖专家经验、隐含判断、日志解释或事后讨论的问题，整理成可描述、可记录、可复用、可培训、可工具化的结构。

从这个角度看，可以形成如下横向关系：

- FMEA 结构化潜在失效模式；
- STPA 结构化安全控制缺陷；
- RCA 结构化事后根因链；
- Process Mining 结构化流程实际运行；
- TPCA / CAE-SDB 结构化目标状态进入前的 false / NG / not ready / pending，并输出控制路径。

这种对标主要说明问题类型和方法论方向，并不表示成熟度已经等同。

FMEA 已经形成表格、评分方式、行业用法、培训体系和软件支持。  
STPA 已经形成控制结构图、不安全控制动作、hazard scenario 等分析步骤。  
Process Mining 已经形成事件日志格式、算法、工具、商业软件和论文生态。  

TPCA / CAE-SDB 当前已经具备的是：

- 问题对象：目标状态进入前的判断不成立；
- 基本分解：C / A / E；
- 判定性质：S / D / B；
- 工程节点：PCN；
- 输出方式：多路径控制；
- 应用方向：case1、case2、case3；
- 产品雏形：MES 协同停滞诊断模块、PLC / HMI 前置判定模板、数字调用前置控制模块。

后续仍需要沉淀标准分析流程、模板、最小数据格式、案例库、PoC 指标、软件工具和外部用户。

因此，TPCA / CAE-SDB 当前更准确的定位是：

面向复杂工程系统状态迁移前置判定问题的结构化工程方法。

## 3. 与既有工业自动化体系的边界

TPCA / CAE-SDB 与状态机、安全控制、报警诊断、AI 诊断和 MES / WCS 调度并行存在。它的作用是补足目标状态进入前的结构化判定层。

### 3.1 与状态机和 SFC 的关系

状态机、SFC 和 PLC 顺序控制负责定义流程、步骤、状态、迁移关系和动作顺序。

它们回答：

系统有哪些状态；  
状态之间如何切换；  
满足什么条件时进入下一状态。

TPCA / CAE-SDB 部署在目标状态进入前，用于展开 transition condition。

它进一步判断：

- 条件信号是否定义并接入；
- 条件信号是否仍然有效；
- 许可来源是否明确；
- 安全、区域、上位、人工确认等许可是否成立；
- 进入目标阶段后执行链是否能够接续；
- 当前状态是否处于边界；
- 不满足时应等待、重识别、回流、降级、禁止进入，还是人工确认。

状态机负责流程推进。  
PCN 负责迁移前判定。

### 3.2 与互锁和安全控制的关系

安全 PLC、安全继电器、安全回路、急停、光栅、安全门和区域进入许可，仍负责现场安全功能。

TPCA / CAE-SDB 接收安全许可状态，并将其映射到 A，也就是 Authority 许可状态。

在 TPCA / CAE-SDB 中，关键 A 可以构成独立必要约束。

当安全门未关闭、光栅被触发、急停被按下、区域许可未成立或上位系统禁止放行时，即使 C 条件成立、E 执行链看似可接续，也不得允许进入目标状态。

边界应保持清楚：

- 安全 PLC、安全继电器和安全回路负责安全功能；
- PCN 接收安全许可状态；
- CAE-SDB 将安全许可作为 A 域进行判定；
- A 不成立时，输出禁止进入、安全锁定、等待许可、人工确认等控制路径；
- PCN 不绕过现场安全系统，也不替代硬件安全回路。

### 3.3 与报警管理和故障诊断的关系

报警管理和故障诊断负责异常检测、报警通知、报警优先级、报警生命周期、事件记录和原因分析。

但复杂自动化系统中，报警本身经常不能直接回答状态迁移问题。

例如：

- 机器人报警，是目标抓取阶段不能进入的源头，还是进入后失败的结果；
- 视觉报警，影响的是 C 条件，还是 C-D 动态有效性；
- 下游报警，是 E 执行链不可接续，还是 A 许可未成立；
- 通信报警，是结构未接入、动态超时，还是边界等待。

TPCA / CAE-SDB 将报警重新放入目标状态进入前的判定结构中，判断报警影响 C、A、E 中哪个变量域，属于 S、D、B 中哪类性质，并进一步输出等待、重采样、重识别、下游协调、禁止进入、安全锁定或人工确认等控制路径。

很多现场并非缺少报警，而是报警与当前目标状态之间的关系没有被结构化表达。TPCA / CAE-SDB 处理的正是这部分关系。

### 3.4 与数据驱动和 AI 诊断的关系

数据驱动和 AI 诊断擅长发现复杂模式，适合处理高维数据、历史日志、异常模式和趋势预测。

但工业 AI 诊断在落地时常遇到两个问题：

第一，原始数据多，但工程语义不够清楚。  
第二，模型能识别异常模式，但现场仍需要判断这个模式对当前控制路径意味着什么。

TPCA / CAE-SDB 不依赖 AI 才能成立。AI 也不直接替代 PCN 的现场判定。

PCN 可以记录：

- 当前状态；
- 目标状态；
- 多源状态信号；
- C/A/E 映射结果；
- S/D/B 判定结果；
- 多路径控制输出；
- 时间戳；
- TraceLogID；
- 人工确认结果；
- 后续是否复发。

这些数据比单纯原始日志更适合后续分析。

AI 可以在 PCN 履历积累之后，用于跨节点模式分析、高频迁移失败节点识别、候选改善方案生成、效果预测、复盘报告生成和调试知识沉淀。

现场控制逻辑、PCN 配置修改、控制边界变更、试行和回退，仍应由工程师、现场负责人和既有管理流程决定。

### 3.5 与 MES / WCS 调度和协同控制的关系

MES / WCS / AGV / AMR 调度系统关注任务、资源、路径和执行主体之间的匹配关系。

它们回答：

谁执行任务；  
何时执行；  
走哪条路径；  
使用哪个资源；  
如何避免冲突；  
如何提高吞吐。

但制造现场经常出现一种问题：调度结果已经生成，任务也已经存在，但系统仍无法进入持续执行状态。

例如：

- MES 有任务，WCS 有记录，但车辆未执行；
- 多台车处于 waiting、blocked、pending；
- 站点不可接收，导致任务滞留；
- 窄通道、自动门、电梯或交接区形成资源竞争；
- 资源锁未释放，后续任务无法进入；
- 车辆集中充电，导致可执行车辆比例下降。

调度系统回答任务如何分配、路径如何规划、资源如何配置。  
TPCA / CAE-SDB 进一步判断调度结果进入执行前，条件、许可和执行链是否成立。

在 case1 中，WCS / 群控 PCN 可以读取 MES、WCS、AGV / AMR、站点、缓存区、路径资源、充电区、安全系统和下游工位等多源状态信号，计算群体未执行比例、群体辅助执行比例、群体能源补给比例、授权未成立比例等指标。

这些指标用于识别协同停滞结构，例如分配约束型、资源竞争型、能源偏在型、授权未成立型和未确定型，并进一步触发任务重分配、资源锁释放、路径复核、通行权协调、限流、改派缓存区、增强记录、人工确认或异常隔离。

TPCA / CAE-SDB 在 MES / WCS 中的定位，是协同停滞的前置判定和结构诊断层。

## 4. TPCA / CAE-SDB 补充的前置判定层

TPCA / CAE-SDB 补充的是目标状态进入前的结构化前置判定层。

其基本处理链条为：

当前状态 / 当前阶段  
→ 目标状态 / 目标阶段  
→ 多源状态信号  
→ C/A/E 状态映射  
→ S/D/B 判定  
→ CAE-SDB 判定结果  
→ 多路径控制输出  
→ 状态记录与履历  
→ 复盘、改善与再部署

C/A/E 表示目标状态进入前的三个状态变量域。

C = Condition，条件状态。  
表示进入目标状态前，现场条件、对象条件、识别条件、数据条件、任务条件、参数条件等是否成立。

A = Authority，许可状态。  
表示系统是否允许进入目标状态，包括安全许可、区域许可、上位许可、人工确认、权限、授权、资源锁、对方设备许可等。

E = Execution Chain，执行链状态。  
表示进入目标状态后，执行链是否能够接续。E 不等于单体 ready。E 包括本体设备、末端执行机构、下游承接、替代路径、回退路径、异常排出路径、结果上传或回写链路等。

S/D/B 表示对 C/A/E 状态变量进行判定时使用的三类性质。

S = Structure，结构完整性。  
判断所需信号、接口、映射关系、许可来源、路径、角色、执行链边界是否已经定义、接入并可观测。

D = Dynamics，动态时序有效性。  
判断信号是否超时、未刷新、抖动、冲突、延迟、不同步、低置信度、许可撤销或处于切换中。

B = Boundary，控制边界。  
判断系统处于未完全确定或边界状态时，下一步应等待、重试、重识别、回流、降级、禁止、人工确认、安全锁定还是异常隔离。

TPCA / CAE-SDB 的最终输出不是单纯原因标签，也不是报警显示，而是控制路径。

典型输出包括：

- 允许进入；
- 等待；
- 重识别；
- 重采样；
- 重定位；
- 回流；
- 异常分流；
- 下游协调；
- 资源释放；
- 降级执行；
- 备用路径；
- 禁止进入；
- 安全锁定；
- 人工确认；
- 异常隔离；
- 增强记录。

PCN 不只完成一次判定，还应记录判定过程。

典型记录内容包括当前状态、目标状态、多源状态信号、信号时间戳、C/A/E 映射结果、S/D/B 判定结果、多路径控制输出、人工确认结果、TraceLogID 和后续处理结果。

多个 PCN 的履历可以形成状态迁移前置判定网络，用于识别高频迁移失败节点、记录控制路径触发历史、分析条件 /许可 / 执行链之间的薄弱点，并支撑现场复盘、模板复用、人员培训和 AI 辅助分析。

## 5. 结论

TPCA / CAE-SDB 应放在既有工业自动化理论体系的边界处理解。

FMEA 结构化潜在失效模式。  
STPA 结构化安全控制缺陷。  
RCA 结构化事后根因链。  
Process Mining 结构化流程实际运行。  
状态机和 SFC 负责定义状态与迁移。  
安全控制负责危险动作禁止和安全功能实现。  
报警管理和故障诊断负责异常检测、通知、记录和原因分析。  
数据驱动与 AI 诊断负责复杂模式识别和辅助改善。  
MES / WCS 调度负责任务、资源、路径和主体分配。  
形式化验证和运行时验证负责模型或运行过程的性质检查。

TPCA / CAE-SDB 处理的是目标状态进入前的运行时前置判定与多路径控制。

它将复杂工程系统中分散存在的状态信号、许可约束、执行链状态、报警、调度结果和人工确认，整理为 PCN 前置控制节点中的 C/A/E 状态映射、S/D/B 判定和多路径控制输出。

其工程定位可以概括为：

复杂工程系统目标状态进入前的运行时前置判定与多路径控制层。

它的价值在于，把原本分散在 PLC 程序、HMI 画面、MES / WCS 日志、机器人控制器、安全许可、调度系统和人工经验中的状态迁移前判断，转化为可配置、可显示、可追溯、可复用的 PCN 判定结构。

TPCA / CAE-SDB 当前仍处于工程方法论早期产品化阶段。后续重点应放在标准分析流程、公开模板、案例库、PoC 指标和工具化实现上，使其从个人方法论逐步发展为外部工程师也能理解、试用和复用的结构化方法。
